# Dental Centre Financial Dashboard — DWH Design

## Overview

Translates the QlikSense load script `Dental_Centre_Financial_Dashboard.md` into a SQL Server
SILVER layer (6 tables) consumed directly by Power BI via Power Query.

**Source systems:**
- D4W (Live) — dental clinic system (SQL, ODBC)
- rpsqlrp01 / paragonreporting — Westfund member system (SQL)
- QVD files — pre-extracted snapshots on prdqs01_atobi
- Excel files — manual budget inputs on prdqs01_atobi

**Architecture decision:** No GOLD view is built. Power BI imports each Silver table
independently via Power Query. Budget Excel files are unpivoted in Power Query and never
pass through SILVER.

---

## Architecture

```
BRONZE (read-only)              SILVER (6 tables)                    Power BI
────────────────────            ──────────────────────────────────   ────────────────────────
D4W Live SQL           ──→      Dental_Financial_Actual      ──→     Power Query
Paragon SQL            ──→      Dental_Payment               ──→     Power Query
QVD files              ──→      Dental_Appointment           ──→     Power Query
                       ──→      Dental_Member_Utilisation    ──→     Power Query
                       ──→      Dental_Dentist_Utilisation   ──→     Power Query
                       ──→      Dental_NPS                   ──→     Power Query

Excel budget files ─────────────────────────────────────────────→   Power Query (Unpivot)
```

### 3. `Dental_Appointment`

**Source values:** `D4W Appointments`, `Chair Utilisation`, `Chair UtilisationTest`, `Patient Retention`

All four Source values are derived from the same `dba.a_appointments` base query.
They are produced as separate NoConcatenate slices in Qlik, each selecting a different
column subset with additional derived fields.

**Data sources:**

| Table / File | Role |
|---|---|
| `dba.a_appointments` | Appointment base (filter: `ref_status IS NULL`, `pat_id > 0`, `pat_id <> 157373`) |
| `dba.patients_hf` | Health fund membership linkage via `Card_No` |
| `ExtractData/Dental/D4W/D4W_CustomField_MbrNo.qvd` | `Member_YorN` flag — `Mbr_No` is a join intermediary only and is NOT selected in `TmpAppointment`'s final NoConcatenate LOAD; only `Member_YorN` reaches Silver |
| `TransformData/Business KPIs Display/DentalPatients_Display.qvd` | First appointment date per patient (`First_Appt_Date`) |

**Filter:** `Appt_Date >= '2023-07-01'` AND `Appt_Date <= TODAY()`

**Inline lookup:**

`AgeCohorts` inline table maps age ranges to `AgeCohort` labels:
`0-17`, `18-29`, `30-44`, `45-59`, `60-74`, `75+`

`Appt_Book_To_DrNumber_MAP` inline table maps `app_book_id → doct_id` (step 1). A second MAP `D4WDrID_MAP` (built from `dba.staff ON staff.member_id`) then resolves `doct_id → ApptBook Dentist Name` (step 2). In SQL the SP must JOIN `dba.a_appointments → inline-lookup → dba.staff ON staff.member_id = doct_id` to derive `Dentist`.

> **Patient Retention column note:** Qlik derives Patient Retention rows from `Appointments_Final` (a reduced intermediate via `AttendedApptBase → Appointments_Sorted → Appointments_Final`). By that stage only `Appt_PatNumber`, `ApptDate`, `NextApptDate`, `DaysTillNextAppt` survive — plus `Source`, `ReturnedWithin12Months`, `Fin_Year`, `Fin_Period`, `MonthYear`, `Weekend`, `Period` added in the Concatenate LOAD. Fields like `Raw_ApptKey`, `Dentist`, `Appt_Duration`, and the four flag columns are dropped earlier in the chain. The SQL SP should source Patient Retention rows from the same `dba.a_appointments` base (mirroring `TmpAppointment_Ret`) to populate only the columns marked Y above.

> **`Period` column format note (IMPORTANT):** `Period` has inconsistent semantics across Source values in Qlik — this is a Qlik bug that the SQL SP should normalise away. `TmpAppointment` (D4W Appointments, line 548) and `TmpAppointment_1` (Chair Utilisation, line 585) both set `Period = [Appt_Period]` (1–12, same value as `Fin_Period`). By contrast, `Chair UtilisationTest` (lines 661–664) and `Patient Retention` (lines 1208–1211) compute `Period` as YYYYPP (e.g. 202501). The SQL SP must **always derive `Period` as `Fin_Year * 100 + Fin_Period` (YYYYPP)** for all four Source values — never copy `Fin_Period` directly as `Period`.

**Note on `Chair UtilisationTest`:** Qlik derives a virtual chair assignment per date using
`IterNo()` (up to 5 chairs). SQL equivalent uses `ROW_NUMBER() OVER (PARTITION BY Date ORDER BY Dentist)`
capped at 5.

> **Columns deliberately excluded from Silver (`TmpNPS` NoConcatenate LOAD):** `Treatments` intermediate holds `notes as [Comment]` (D4W path) and `Feedback as Comment` (Qualtrics QVD path) — raw patient complaint/compliment text and open-text survey feedback — which Qlik drops before Silver. The Qualtrics path also loads `ResponseId` (line 832 of Qlik script) as a unique survey response identifier, which is likewise not selected into `TmpNPS` and does not reach Silver. Both exclusions are intentional. If free-text feedback is needed in future, add `Comment NVARCHAR(MAX)` to `create_table_Dental_NPS.sql` and include `notes` (D4W) / `Feedback` (Qualtrics) in the SP INSERT. If deduplication by response is needed, add `ResponseId NVARCHAR(100)` and populate it for the Qualtrics path only.
